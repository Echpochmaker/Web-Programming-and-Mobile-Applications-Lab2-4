# Лабораторные работы №2-6: REST API для системы онлайн-тестирования с авторизацией, кешированием, документированием и MongoDB

Данный проект представляет собой RESTful веб-сервис для управления тестами, вопросами и вариантами ответов с полноценной системой аутентификации, авторизации, кешированием на Redis, автоматически сгенерированной документацией OpenAPI (Swagger) и документоориентированной базой данных MongoDB.

Реализован на **FastAPI** с использованием **Beanie ODM** (MongoDB), **Redis** для кеширования. Вся инфраструктура запускается через **Docker Compose**, что обеспечивает лёгкое развёртывание и изоляцию сервисов.

## 🎯 Основные возможности по лабораторным работам

### Лабораторная работа №2 — CRUD ресурсов

- Полноценный CRUD для сущности `Test` (тест), `Question` (вопрос) и `AnswerOption` (вариант ответа)
- Мягкое удаление (soft delete) — записи помечаются удалёнными, но не стираются из базы
- Пагинация при получении списка тестов (параметры `page` и `limit`)
- Валидация входящих данных через Pydantic-схемы (DTO)
- Модульная архитектура: контроллеры, сервисы, модели, DTO разделены по папкам

### Лабораторная работа №3 — Аутентификация и авторизация

- Регистрация новых пользователей с хешированием паролей (bcrypt + уникальная соль)
- Вход в систему с выдачей JWT токенов (Access и Refresh)
- Безопасное хранение токенов в HttpOnly cookies
- Refresh токены с возможностью отзыва сессий
- Защита эндпоинтов — доступ только для авторизованных пользователей
- Проверка владения ресурсами (пользователь может редактировать/удалять только свои тесты)
- Выход из текущей сессии (`/logout`) и из всех сессий (`/logout-all`)
- Эндпоинт `/whoami` для проверки статуса авторизации
- Восстановление пароля через email (токен выводится в консоль Docker)
- Интеграция с OAuth 2.0 (Yandex ID) — ручная реализация без готовых библиотек
- Защита от CSRF с использованием параметра `state`

### Лабораторная работа №4 — Автоматическая документация OpenAPI/Swagger

- **Автоматическая генерация** спецификации OpenAPI на основе кода (Code-First подход)
- **Условный запуск** — документация доступна только в режиме разработки (`development`)
- **Интерактивный Swagger UI** для тестирования всех эндпоинтов
- **Подробные описания** всех эндпоинтов с примерами запросов и ответов
- **Схемы безопасности**:
  - `bearerAuth` — для тестирования с JWT токеном
  - `cookieAuth` — для работы с HttpOnly cookies
- **Примеры ответов** для всех HTTP статусов (200, 201, 400, 401, 403, 404)
- **Документирование DTO** — каждое поле Pydantic-модели содержит описание и пример значения

### Лабораторная работа №5 — Кеширование данных и управление сессиями с Redis

- **Кеширование списков тестов** — повторные запросы страниц тестов возвращаются из Redis, а не из MongoDB
- **Кеширование профиля пользователя** — данные `/auth/whoami` кешируются на 5 минут
- **JTI (JWT ID) для Access токенов** — каждый токен получает уникальный идентификатор, который хранится в Redis
- **Мгновенный отзыв токенов** — при логауте JTI удаляется, и токен становится недействительным до истечения срока жизни
- **Инвалидация кеша** — любая операция записи (POST, PUT, PATCH, DELETE) очищает связанные кешированные данные
- **TTL (Time To Live)** — все ключи в Redis имеют ограниченное время жизни
- **Защита паролем** — Redis защищён паролем, который хранится в `.env` файле
- **Отдельный модуль кеширования** (`app/core/cache.py`) с методами `get`, `set`, `delete`, `delete_pattern`

### Лабораторная работа №6 — Миграция с PostgreSQL на MongoDB

- **Замена реляционной СУБД на документоориентированную** — PostgreSQL заменён на MongoDB
- **ObjectId вместо автоинкремента** — идентификаторы генерируются драйвером
- **Встраивание (Embedding)** — вопросы и ответы вложены в документ теста (нет отдельных коллекций)
- **Гибкая схема** — валидация на уровне приложения через Beanie ODM
- **Сохранение всей функциональности** — CRUD, пагинация, мягкое удаление, JWT, Redis, Swagger работают с MongoDB

- Swagger UI: `http://localhost:4200/api/docs` (только в режиме development)
- Веб-интерфейс: `http://localhost:4200`

5. **Остановка**:
```bash
docker-compose down
```

## ⚡ Кеширование данных с Redis — ЛР №5

### Общая схема работы

```
┌─────────────┐     Cache Miss       ┌─────────────┐
│   Клиент    │ ──────────────────►  │  Redis      │
│  (браузер)  │                      │  (кеш)      │
└─────────────┘                      └─────────────┘
       │                                    │
       │                                    │ Cache Miss
       │                                    ▼
       │                             ┌─────────────┐
       │                             │   MongoDB   │
       │                             │    (БД)     │
       │                             └─────────────┘
       │                                     │
       │◄────────────────────────────────────┘
       │           Cache HIT (данные из Redis)
       │
       ▼
   Получен ответ
```

### Ключи кеша и TTL

| Префикс ключа | Описание | TTL |
|---------------|----------|-----|
| `testing:tests:list:page:{page}:limit:{limit}` | Кешированные списки тестов | 300 сек (5 мин) |
| `testing:users:profile:{user_id}` | Кешированный профиль пользователя | 300 сек (5 мин) |
| `testing:auth:user:{user_id}:access:{jti}` | JTI активного Access токена | 900 сек (15 мин) |

### Механизм JTI (JWT ID)

При входе в систему:
1. Генерируется уникальный JTI (`uuid.uuid4()`)
2. JTI сохраняется в Redis с TTL = время жизни Access токена (15 минут)
3. JTI добавляется в payload JWT токена

При каждом запросе к защищённому эндпоинту:
1. Из JWT токена извлекается JTI
2. Проверяется наличие JTI в Redis
3. Если JTI отсутствует → токен считается отозванным (`401 Token revoked`)

При логауте:
1. Все JTI текущего пользователя удаляются из Redis
2. Токен становится недействительным мгновенно (до истечения TTL)

### Инвалидация кеша

| Операция | Действие с кешем |
|----------|------------------|
| `POST /tests/` | Удаление всех кешированных списков (`testing:tests:list:*`) |
| `PUT /tests/{id}` | Удаление всех кешированных списков |
| `PATCH /tests/{id}` | Удаление всех кешированных списков |
| `DELETE /tests/{id}` | Удаление всех кешированных списков |
| `POST /auth/login` | Удаление кеша профиля пользователя |
| `POST /auth/logout` | Удаление JTI и кеша профиля |

### Проверка работы кеша через Redis CLI

**Подключение к Redis (с паролем):**
```bash
docker exec -it testing_redis redis-cli --pass redis_secure_password_change_in_prod
```

**Полезные команды:**
```bash
# Просмотр всех ключей приложения
KEYS testing:*

# Получение кешированного списка тестов
GET "testing:tests:list:page:1:limit:10"

# Проверка TTL (времени жизни)
TTL "testing:tests:list:page:1:limit:10"

# Просмотр JTI активных сессий
KEYS testing:auth:user:*

# Удаление всех ключей (для тестов)
FLUSHDB
```

## 🌐 Веб-интерфейс

Проект включает простой веб-интерфейс, доступный по адресу `http://localhost:4200`.

### Функционал веб-интерфейса

- **Мои тесты** (главная страница)
  - Просмотр списка своих тестов
  - Создание нового теста
  - Редактирование теста (добавление/удаление вопросов и ответов)
  - Удаление теста (мягкое удаление)
  - Пагинация

- **Пройти тест**
  - Просмотр доступных тестов (чужие тесты)
  - Поиск по названию
  - Прохождение теста с сохранением ответов
  - Просмотр результата с детализацией по вопросам

- **Результаты**
  - Просмотр всех пройденных тестов
  - Отображение процента правильных ответов
  - Детальный просмотр каждого вопроса с правильным/неправильным ответом

- **Авторизация**
  - Регистрация нового пользователя
  - Вход по email/паролю
  - Вход через Яндекс ID (OAuth 2.0)
  - Восстановление пароля
  - Выход из системы

### Тёмная тема

В правом верхнем углу находится виджет-меню (иконка 👤), который позволяет:
- Переключить тему оформления (светлая/тёмная)
- Выбор темы сохраняется в `localStorage` и применяется на всех страницах


## 🔒 Безопасность данных в кеше (ЛР №5)

| Требование | Реализация |
|------------|------------|
| Запрещено хранение паролей в кеше | В кеше только профиль (без `password_hash`) и JTI |
| Запрещено хранение полных токенов | Хранится только JTI (идентификатор) |
| Префиксы для ключей | Все ключи имеют префикс `testing:` |
| TTL для всех ключей | 300 сек для данных, 900 сек для JTI |
| Защита Redis паролем | `redis-server --requirepass ${REDIS_PASSWORD}` |


## 🐳 Обновлённый `docker-compose.yml` (ЛР №6)

```yaml
services:
  mongo:
    image: mongo:6
    container_name: testing_mongo
    restart: unless-stopped
    environment:
      MONGO_INITDB_ROOT_USERNAME: ${DB_USER}
      MONGO_INITDB_ROOT_PASSWORD: ${DB_PASSWORD}
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db
    networks:
      - testing_network
    healthcheck:
      test: ["CMD", "mongosh", "--eval", "db.adminCommand('ping')"]
      interval: 10s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    container_name: testing_redis
    restart: unless-stopped
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    networks:
      - testing_network
    command: redis-server --requirepass ${REDIS_PASSWORD} --appendonly yes
    healthcheck:
      test: ["CMD", "redis-cli", "--pass", "${REDIS_PASSWORD}", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  app:
    build: .
    container_name: testing_app
    restart: unless-stopped
    environment:
      MONGO_URI: ${MONGO_URI}
      REDIS_HOST: redis
      REDIS_PORT: 6379
      REDIS_PASSWORD: ${REDIS_PASSWORD}
      CACHE_TTL_DEFAULT: 300
      YANDEX_CLIENT_ID: ${YANDEX_CLIENT_ID}
      YANDEX_CLIENT_SECRET: ${YANDEX_CLIENT_SECRET}
      YANDEX_CALLBACK_URL: ${YANDEX_CALLBACK_URL}
      JWT_ACCESS_SECRET: ${JWT_ACCESS_SECRET}
      JWT_REFRESH_SECRET: ${JWT_REFRESH_SECRET}
      JWT_ACCESS_EXPIRATION: 15m
      JWT_REFRESH_EXPIRATION: 7d
      APP_ENV: development
    ports:
      - "4200:4200"
    depends_on:
      mongo:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - testing_network
    command: uvicorn app.main:app --host 0.0.0.0 --port 4200 --reload

volumes:
  mongo_data:
  redis_data:

networks:
  testing_network:
```
